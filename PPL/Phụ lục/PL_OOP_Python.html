Mở đầu

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Python là một ngôn ngữ hỗ trợ lập trình hướng đối tượng rất linh hoạt. Mọi giá trị trong Python đều là đối tượng (object): số nguyên, chuỗi, danh sách, hàm, lớp và cả module đều là đối tượng. Vì vậy, khi học OOP trong Python, ta không chỉ học cách viết class, mà còn cần hiểu cách Python biểu diễn đối tượng, cách phương thức nhận đối tượng hiện hành qua self, cách Python xử lý đóng gói bằng quy ước, cách kế thừa hoạt động, cách thứ tự phân giải phương thức (Method Resolution Order, MRO) được tính, và vì sao Python thường gắn với kiểu vịt (duck typing).

Lớp và đối tượng trong Python

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Trong Python, lớp (class) là khuôn mẫu mô tả cấu trúc và hành vi chung của các đối tượng. Đối tượng (object) là một thực thể cụ thể được tạo ra từ lớp tại thời điểm chạy. Ví dụ:

Python – lớp Employee
class Employee:
    emp_count = 0
    def __init__(self, name, salary):
        self.name = name
        self.salary = salary
        Employee.emp_count += 1

Tạo hai đối tượng:

Python
e1 = Employee("Nam", 30)
e2 = Employee("Lan", 35)

Trong đoạn mã trên, Employee là lớp, còn e1 và e2 là hai đối tượng khác nhau. Mỗi đối tượng có định danh (identity) riêng và trạng thái (state) riêng.

Python – kết quả
print(e1.name, e1.salary)  # Nam 30
print(e2.name, e2.salary)  # Lan 35
print(e1 is e2)           # False

Điểm quan trọng là hai đối tượng có thể có cùng dữ liệu nhưng vẫn là hai thực thể khác nhau:

Python – định danh độc lập với trạng thái
a = Employee("An", 40)
b = Employee("An", 40)
print(a.name == b.name)    # True
print(a.salary == b.salary) # True
print(a is b)              # False

Hai đối tượng a và b có cùng trạng thái nhưng không có cùng định danh. Điều này phản ánh một nguyên lý quan trọng của OOP: định danh của đối tượng độc lập với trạng thái của đối tượng.

Instance attribute và class attribute

Trong Python, thuộc tính có thể nằm ở hai mức: thuộc tính đối tượng (instance attribute) và thuộc tính lớp (class attribute).

Python – class attribute vs instance attribute
class Employee:
    emp_count = 0          # class attribute – chia sẻ toàn lớp
    def __init__(self, name):
        self.name = name   # instance attribute – riêng từng đối tượng
        Employee.emp_count += 1

e1 = Employee("Nam")
e2 = Employee("Lan")
print(e1.name)          # Nam
print(e2.name)          # Lan
print(Employee.emp_count) # 2
Cần cẩn thận khi dùng thuộc tính lớp với dữ liệu có thể thay đổi, chẳng hạn danh sách. Danh sách là class attribute sẽ được chia sẻ bởi mọi đối tượng — một thay đổi ảnh hưởng đến tất cả.
Python – lỗi phổ biến với class attribute là list
class Team:
    members = []                           # DÙNG CHUNG!
    def add(self, name): self.members.append(name)

t1 = Team(); t2 = Team()
t1.add("An"); t2.add("Binh")
print(t1.members)  # ['An', 'Binh'] ← không mong đợi!
print(t2.members)  # ['An', 'Binh']

# Sửa: tạo list riêng trong __init__
class Team:
    def __init__(self): self.members = []
    def add(self, name): self.members.append(name)

t1 = Team(); t2 = Team()
t1.add("An"); t2.add("Binh")
print(t1.members)  # ['An']
print(t2.members)  # ['Binh']

Phương thức và self trong Python

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Trong Python, phương thức (method) là hàm được định nghĩa bên trong lớp và thường thao tác trên đối tượng hiện hành. Tham số đầu tiên của phương thức thường được đặt tên là self.

Python – phương thức và self
class Employee:
    def __init__(self, name, salary):
        self.name = name
        self.salary = salary
    def display(self):
        print(self.name, self.salary)

e = Employee("Nam", 30)
e.display()  # Nam 30

Lời gọi e.display() về bản chất tương đương với Employee.display(e). Nói cách khác, đối tượng e được truyền vào tham số đầu tiên self. Python làm rõ đối tượng nhận lời gọi bằng self, trong khi nhiều ngôn ngữ khác dùng từ khóa ẩn như this.

Python – mỗi đối tượng có trạng thái riêng
class Counter:
    def __init__(self): self.value = 0
    def inc(self): self.value += 1

c1 = Counter(); c2 = Counter()
c1.inc(); c1.inc(); c2.inc()
print(c1.value)  # 2
print(c2.value)  # 1

Nếu quên khai báo self, phương thức sẽ lỗi khi được gọi qua đối tượng, vì Python tự động truyền đối tượng vào tham số đầu tiên nhưng phương thức không khai báo tham số đó.

Ba loại phương thức

Trong Python, ngoài phương thức đối tượng thông thường, còn có phương thức lớp (class method) và phương thức tĩnh (static method).

Python – classmethod và staticmethod
class Employee:
    emp_count = 0
    def __init__(self, name):
        self.name = name
        Employee.emp_count += 1

    @classmethod
    def count(cls):
        return cls.emp_count

class Math:
    @staticmethod
    def square(x):
        return x * x

print(Employee.count())  # số nhân viên đã tạo
print(Math.square(5))     # 25
Loại phương thứcTham số đầu tiênDùng khi nào
Phương thức đối tượngselfCần truy cập trạng thái của đối tượng
Phương thức lớpclsCần truy cập hoặc tạo đối tượng dựa trên lớp
Phương thức tĩnhKhông có tham số tự độngHàm tiện ích đặt trong lớp cho có tổ chức

Trong bối cảnh học OOP cơ bản, phương thức đối tượng với self là quan trọng nhất, vì nó thể hiện rõ ý tưởng: đối tượng nhận lời gọi và tự xử lý dữ liệu của chính nó.

Quy ước đóng gói trong Python

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Trong nhiều ngôn ngữ như Java hoặc C++, đóng gói (encapsulation) thường đi kèm các mức truy cập như public, private, protected. Python không thực thi kiểm soát truy cập chặt chẽ theo cách đó. Thay vào đó, Python dựa nhiều vào quy ước đặt tên và một cơ chế gọi là đổi tên nội bộ (name mangling).

Dạng tênÝ nghĩa
nameCông khai theo quy ước
_nameDành cho nội bộ, không nên truy cập trực tiếp từ bên ngoài
__nameĐược đổi tên nội bộ, tạo hiệu ứng gần giống riêng tư
__name__Tên đặc biệt của Python, không nên tự đặt tùy tiện

Ví dụ thông thường với thuộc tính công khai:

Python – thuộc tính công khai (không an toàn)
class Account:
    def __init__(self, balance):
        self.balance = balance
    def deposit(self, amount):
        self.balance += amount

acc = Account(100)
acc.balance = -1000  # trạng thái không hợp lệ!

Một thiết kế tốt hơn là dùng thuộc tính nội bộ và cung cấp phương thức kiểm soát:

Python – thuộc tính nội bộ _balance
class Account:
    def __init__(self, balance):
        if balance < 0:
            raise ValueError("balance must be non-negative")
        self._balance = balance
    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self._balance += amount
    def get_balance(self):
        return self._balance

acc = Account(100)
acc.deposit(50)
print(acc.get_balance())  # 150

Name mangling với __name

Nếu dùng hai dấu gạch dưới, Python sẽ đổi tên thuộc tính. Ví dụ:

Python – name mangling
class Account:
    def __init__(self, balance):
        self.__balance = balance
    def get_balance(self):
        return self.__balance

acc = Account(100)
# print(acc.__balance)  ← lỗi AttributeError
print(acc.__dict__)
# {'_Account__balance': 100}

# Vẫn có thể truy cập nếu cố tình:
print(acc._Account__balance)  # 100
Cơ chế __name không phải bảo mật tuyệt đối. Mục tiêu chính là tránh xung đột tên trong kế thừa và báo hiệu rằng thuộc tính không nên được truy cập trực tiếp.

Property – cách Pythonic hơn

Một cách Pythonic hơn để kiểm soát truy cập là dùng property:

Python – @property
class Account:
    def __init__(self, balance):
        self._balance = balance

    @property
    def balance(self):
        return self._balance

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("amount must be positive")
        self._balance += amount

acc = Account(100)
print(acc.balance)  # 100 – đọc được
acc.deposit(50)
print(acc.balance)  # 150
# acc.balance = -100  ← lỗi nếu không có setter

Python không ép đóng gói bằng rào cản cứng như Java, nhưng cung cấp quy ước và công cụ để người lập trình thiết kế giao diện ổn định, hạn chế phụ thuộc vào cài đặt bên trong.

Kế thừa và MRO trong Python

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Kế thừa đơn và ghi đè phương thức

Python hỗ trợ kế thừa (inheritance) và cả đa kế thừa (multiple inheritance). Khi một lớp kế thừa từ lớp khác, nó có thể dùng lại thuộc tính và phương thức của lớp cha, đồng thời có thể định nghĩa thêm hoặc ghi đè phương thức. Ví dụ kế thừa đơn:

Hình 1. Kế thừa đơn: Lecturer kế thừa từ Employee.
Python – kế thừa đơn
class Employee:
    def get_info(self):
        return "Employee"

class Lecturer(Employee):
    def teach(self):
        return "Teaching"

lec = Lecturer()
print(lec.get_info())  # Employee – kế thừa từ cha
print(lec.teach())     # Teaching

Nếu lớp con định nghĩa phương thức trùng tên với lớp cha, đó là ghi đè phương thức (method overriding):

Python – method overriding
class Employee:
    def get_info(self): return "Employee"

class Lecturer(Employee):
    def get_info(self): return "Lecturer"

print(Lecturer().get_info())  # Lecturer

Câu hỏi quan trọng là: khi một phương thức được gọi, Python tìm phương thức đó ở đâu và theo thứ tự nào? Câu trả lời nằm ở thứ tự phân giải phương thức (Method Resolution Order, MRO). MRO là thứ tự các lớp mà Python sẽ lần lượt tìm kiếm khi cần phân giải một thuộc tính hoặc phương thức.

Hình 2. Mô phỏng tìm phương thức: Python duyệt lần lượt các lớp trong MRO cho đến khi tìm thấy tên cần phân giải.
Python – xem MRO
class A:
    def f(self): return "A"
class B(A): pass
class C(B): pass
print(C.__mro__)
# (<class 'C'>, <class 'B'>, <class 'A'>, <class 'object'>)
print(C().f())  # A – tìm thấy f trong A
Hình 3. Thứ tự tìm kiếm MRO của C(B(A)): C → B → A → object. C không có f, B không có f, A có f.

Đa kế thừa và bài toán kim cương

Trong kế thừa đơn, thứ tự tìm kiếm thường dễ hiểu. Nhưng trong đa kế thừa, một lớp có thể có nhiều lớp cha.

Hình 4. Đa kế thừa đơn giản: C(A, B).
Python – đa kế thừa C(A, B)
class A: pass
class B: pass
class C(A, B): pass
print(" -> ".join(cls.__name__ for cls in C.__mro__))
# C -> A -> B -> object

Vấn đề trở nên khó hơn khi các lớp cha lại có tổ tiên chung — đây là bài toán kim cương.

Hình 5. Bài toán kim cương: D(B, C) với B và C cùng kế thừa từ A.
Python – bài toán kim cương
class A:
    def f(self): return "A"
class B(A):
    def f(self): return "B"
class C(A):
    def f(self): return "C"
class D(B, C): pass

print(" -> ".join(cls.__name__ for cls in D.__mro__))
# D -> B -> C -> A -> object
print(D().f())  # B

# Đổi thứ tự kế thừa → kết quả thay đổi
class D2(C, B): pass
print(D2().f())  # C
Thứ tự lớp cha trong khai báo ảnh hưởng đến MRO, và do đó ảnh hưởng đến phương thức nào được chọn khi có xung đột tên.

Giải thuật C3

Python hiện đại dùng tuyến tính hóa C3 (C3 linearization) để tính MRO. Mục tiêu của C3 là tạo ra một danh sách tuyến tính các lớp cần tìm kiếm, đồng thời bảo đảm ba tính chất quan trọng:

Tính chấtÝ nghĩa
Lớp con đứng trước lớp chaNếu B kế thừa từ A, thì B phải xuất hiện trước A
Giữ thứ tự khai báo lớp chaNếu D(B, C), thì B phải được ưu tiên trước C
Nhất quán đơn điệuThứ tự đã có trong lớp cha không bị đảo lộn tùy tiện ở lớp con

Công thức MRO của một lớp C có các lớp cha B1, B2, ..., Bn là:

Công thức C3
MRO(C) = [C] + merge(MRO(B1), MRO(B2), ..., MRO(Bn), [B1, B2, ..., Bn])

Hàm merge hoạt động theo ý tưởng: lặp cho đến khi tất cả danh sách rỗng:

  1. Chọn phần tử đầu tiên của một danh sách nếu phần tử đó không xuất hiện trong phần đuôi của bất kỳ danh sách nào khác.
  2. Đưa phần tử được chọn vào kết quả.
  3. Xóa phần tử đó khỏi đầu các danh sách có chứa nó.
  4. Nếu không chọn được phần tử nào → MRO không hợp lệ.

Lưu ý: "phần đuôi" = các phần tử đứng sau phần tử đầu tiên của một danh sách.

Ví dụ với hình kim cương (Hình 5):

Tính MRO bằng C3 – bài toán kim cương
MRO(A) = [A, object]
MRO(B) = [B, A, object]
MRO(C) = [C, A, object]
MRO(D) = [D] + merge([B, A, object], [C, A, object], [B, C])
BướcCác danh sách đang xétChọnLý do
1[B, A, object], [C, A, object], [B, C]BB không nằm trong đuôi danh sách nào
2[A, object], [C, A, object], [C]CA nằm trong đuôi [C, A, object], nên chưa chọn A; chọn C
3[A, object], [A, object]AA không còn nằm trong đuôi danh sách nào
4[object], [object]objectCòn lại object
Python – kiểm tra
print([cls.__name__ for cls in D.__mro__])
# ['D', 'B', 'C', 'A', 'object']

Ví dụ MRO phức tạp hơn

Xét phân cấp sau:

Hình 6. Phân cấp kế thừa phức tạp dùng minh họa giải thuật C3.
Python – F(D, E) với D(A,B) và E(B,C)
class O: pass
class A(O): pass
class B(O): pass
class C(O): pass
class D(A, B): pass
class E(B, C): pass
class F(D, E): pass
Tính MRO
MRO(D) = [D, A, B, O, object]
MRO(E) = [E, B, C, O, object]
MRO(F) = [F] + merge([D, A, B, O, object], [E, B, C, O, object], [D, E])
BướcCác đầu danh sáchChọnGiải thích
1D, E, DDD không nằm trong đuôi danh sách nào
2A, E, EAA không nằm trong đuôi danh sách nào
3B, E, EEB nằm trong đuôi MRO(E), nên chưa chọn B; chọn E
4B, BBB hợp lệ
5O, CCO nằm trong đuôi [C, O, object], nên chọn C
6O, OOO hợp lệ
7object, objectobjectCòn lại object
Python – kiểm tra
print([cls.__name__ for cls in F.__mro__])
# ['F', 'D', 'A', 'E', 'B', 'C', 'O', 'object']

Ví dụ này cho thấy MRO không đơn giản là duyệt theo chiều sâu từ trái sang phải. Python dùng C3 để bảo toàn thứ tự ưu tiên cục bộ và quan hệ kế thừa đã có trong các lớp cha.

Khi MRO không hợp lệ

Không phải mọi phân cấp đa kế thừa đều hợp lệ. Nếu các lớp cha yêu cầu thứ tự mâu thuẫn nhau, Python không thể tạo MRO.

Hình 7. Phân cấp mâu thuẫn: X(A, B) và Y(B, A) không thể cùng được thỏa mãn trong Z(X, Y).
Python – MRO không hợp lệ
class A: pass
class B: pass
class X(A, B): pass
class Y(B, A): pass
class Z(X, Y): pass  # TypeError!
# TypeError: Cannot create a consistent method resolution order
LớpThứ tự yêu cầu
X(A, B)A trước B
Y(B, A)B trước A
Z(X, Y)Phải tôn trọng cả hai
Kết quảMâu thuẫn, MRO không tồn tại

Đây là lý do Python không chỉ chọn đại một thứ tự, mà kiểm tra tính nhất quán của phân cấp kế thừa.

MRO và super()

Trong Python, super() không đơn giản có nghĩa là "gọi lớp cha trực tiếp". Chính xác hơn, super() gọi phương thức tiếp theo trong MRO.

Hình 8. super() đi theo MRO của đối tượng, không chỉ theo lớp cha trực tiếp.
Python – super() theo MRO
class A:
    def f(self): print("A")
class B(A):
    def f(self): print("B"); super().f()
class C(A):
    def f(self): print("C"); super().f()
class D(B, C):
    def f(self): print("D"); super().f()

D().f()
# D – B – C – A
# MRO: D → B → C → A → object
# Trong B.f, super().f() gọi C.f (không phải A.f trực tiếp!)
Điểm quan trọng: super() đi theo MRO của đối tượng hiện hành, không chỉ theo lớp cha ghi trong định nghĩa lớp hiện tại.

Kế thừa hợp tác

Để đa kế thừa hoạt động tốt, các lớp nên được viết theo kiểu kế thừa hợp tác (cooperative inheritance). Nghĩa là mỗi lớp làm phần việc của mình rồi gọi super() để chuyển tiếp cho lớp tiếp theo trong MRO.

Python – kế thừa hợp tác C(A, B)
class Base:
    def process(self): print("Base")
class A(Base):
    def process(self): print("A"); super().process()
class B(Base):
    def process(self): print("B"); super().process()
class C(A, B):
    def process(self): print("C"); super().process()

C().process()
# C → A → B → Base
# MRO: C → A → B → Base → object

Nếu một lớp không gọi super(), chuỗi xử lý sẽ bị ngắt. Vì vậy, khi thiết kế mixin hoặc đa kế thừa trong Python, nên thống nhất cách dùng super() để các lớp có thể phối hợp với nhau.

Mixin và MRO

Mixin là lớp nhỏ cung cấp một năng lực bổ sung, thường không được dùng độc lập. Mixin rất phổ biến trong Python vì Python hỗ trợ đa kế thừa và MRO rõ ràng.

Python – Mixin
class JsonMixin:
    def to_json(self): return self.__dict__
class LogMixin:
    def log(self): print("LOG:", self.__dict__)
class Student(JsonMixin, LogMixin):
    def __init__(self, name, gpa):
        self.name = name; self.gpa = gpa

s = Student("An", 3.8)
print(s.to_json())  # {'name': 'An', 'gpa': 3.8}
s.log()            # LOG: {'name': 'An', 'gpa': 3.8}
print([cls.__name__ for cls in Student.__mro__])
# ['Student', 'JsonMixin', 'LogMixin', 'object']

Nếu hai mixin có cùng tên phương thức, MRO quyết định phương thức nào được chọn:

Python – thứ tự mixin ảnh hưởng đến kết quả
class A:
    def show(self): print("A")
class B:
    def show(self): print("B")
class C(A, B): pass
class D(B, A): pass
C().show()  # A – C(A,B) ưu tiên A
D().show()  # B – D(B,A) ưu tiên B
Khi dùng nhiều mixin, thứ tự lớp cha trong khai báo là một phần của thiết kế, không phải chi tiết ngẫu nhiên.

Tổng kết về MRO

MRO trong Python là cơ chế trung tâm để hiểu kế thừa, đặc biệt là đa kế thừa. Khi gọi obj.method(), Python tìm phương thức theo MRO của lớp động của obj.

Hiểu MRO giúp người lập trình Python tránh nhiều lỗi khó phát hiện trong các hệ thống dùng kế thừa phức tạp, mixin hoặc framework lớn.

Kiểu vịt và đa hình trong Python

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Python thường được mô tả là ngôn ngữ sử dụng kiểu vịt (duck typing). Ý tưởng là: nếu một đối tượng có hành vi cần thiết tại thời điểm chạy, ta có thể dùng nó, không nhất thiết phải kiểm tra nó thuộc lớp nào. Câu nói quen thuộc là: nếu một thứ đi như vịt và kêu như vịt, ta có thể xem nó như vịt.

Python – duck typing
class Circle:
    def draw(self): print("Draw circle")
class Button:
    def draw(self): print("Draw button")

def render(x): x.draw()

render(Circle())  # Draw circle
render(Button())  # Draw button

Circle và Button không cùng kế thừa từ lớp cha chung. Tuy nhiên, cả hai đều có phương thức draw, nên hàm render có thể làm việc với cả hai. Điều này khác với phong cách định kiểu theo tên trong Java.

Nếu truyền vào một đối tượng không có draw, lỗi sẽ xuất hiện khi chạy:

Python – AttributeError
class Student: pass
render(Student())
# AttributeError: 'Student' object has no attribute 'draw'

Kiểu vịt làm chương trình linh hoạt, nhưng cũng đòi hỏi người lập trình cẩn thận hơn. Phong cách thường được khuyến khích trong Python là EAFP (Easier to Ask Forgiveness than Permission):

Python – EAFP và hasattr
# Cách EAFP – thử và xử lý lỗi
def render(x):
    try:
        x.draw()
    except AttributeError:
        raise TypeError("object must provide draw()")

# Cách LBYL – kiểm tra trước
def render(x):
    if hasattr(x, "draw"):
        x.draw()
    else:
        raise TypeError("object must provide draw()")

Python cũng có thể kết hợp duck typing với lớp cơ sở trừu tượng (abstract base class) hoặc giao thức (protocol) khi cần mô tả rõ năng lực mong đợi:

Python – Abstract Base Class và Protocol
from abc import ABC, abstractmethod
class Drawable(ABC):
    @abstractmethod
    def draw(self): pass

class Circle(Drawable):
    def draw(self): print("Draw circle")

# Dùng Protocol – kiểu duck typing có kiểm tra tĩnh
from typing import Protocol
class Drawable(Protocol):
    def draw(self) -> None: ...

def render(x: Drawable) -> None:
    x.draw()

class Button:
    def draw(self) -> None:
        print("Draw button")

render(Button())  # hợp lệ – Button có phương thức draw

Protocol giúp công cụ kiểm tra kiểu tĩnh hiểu rằng render cần một đối tượng có phương thức draw, nhưng vẫn giữ phong cách linh hoạt của Python. Một lớp không cần khai báo kế thừa từ Drawable; chỉ cần có phương thức draw phù hợp là đủ.

Python hỗ trợ đa hình theo cách rất linh hoạt. Trong các chương trình nhỏ, duck typing giúp mã ngắn gọn và tự nhiên. Trong các hệ thống lớn, có thể kết hợp duck typing với type hints, abstract base classes hoặc protocols để tăng khả năng kiểm tra và tài liệu hóa giao diện mong đợi.

Tóm tắt phụ lục

CO3005 · Phụ lục: OOP trong Python · Trần Ngọc Bảo Duy

Python hỗ trợ OOP bằng một mô hình linh hoạt và thực dụng. Lớp mô tả cấu trúc và hành vi chung, đối tượng là thực thể cụ thể tại thời điểm chạy, phương thức nhận đối tượng hiện hành qua self, và thuộc tính có thể nằm ở mức đối tượng hoặc mức lớp.

Python không áp đặt đóng gói bằng cơ chế truy cập cứng như Java hoặc C++, mà dựa vào quy ước đặt tên, name mangling và property. Điều này làm Python linh hoạt nhưng cũng đòi hỏi người lập trình tôn trọng giao diện của lớp.

Trong kế thừa, đặc biệt là đa kế thừa, MRO là cơ chế cốt lõi. Python dùng giải thuật C3 để xác định thứ tự tìm phương thức, bảo đảm lớp con đứng trước lớp cha, giữ thứ tự khai báo lớp cha và tránh các thứ tự kế thừa mâu thuẫn. super() trong Python đi theo MRO chứ không đơn giản là gọi lớp cha trực tiếp.

Cuối cùng, Python thường dùng duck typing: đối tượng được sử dụng dựa trên hành vi mà nó cung cấp tại thời điểm chạy. Cách tiếp cận này giúp Python rất linh hoạt trong thực hành OOP, đồng thời cho thấy rằng lập trình hướng đối tượng không chỉ là quan hệ lớp cha – lớp con, mà còn là cách các đối tượng cung cấp hành vi và tương tác với nhau.
